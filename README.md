# Training Coach

Локальный MVP веб-приложения для планирования тренировочного процесса по данным из спортивных сервисов и часов.

## Что уже есть

- Импорт тренировок из `TCX`, `GPX`, `CSV`, `JSON` независимо от источника экспорта.
- Ручное добавление тренировки.
- Профиль с целевой дистанцией: 5 км, 10 км, 21 км или 42 км.
- Локальное хранение данных в SQLite через backend.
- Резервное хранение в браузере через `localStorage` для миграции и offline fallback.
- Расчет нагрузки в TRIMP: готовая кардионагрузка из импорта используется напрямую, иначе TRIMP считается по длительности и пульсу.
- Обзор нагрузки за 7 и 28 дней.
- Черновой недельный план с понедельника по воскресенье: восстановление/отдых в понедельник, интервалы, темпо, кроссы и длительная в воскресенье.
- Формирование плана через OpenRouter API при запущенном локальном backend.
- Готовый текстовый контекст для ручной передачи в LLM.

## Как запустить

### Вариант без ИИ

Откройте файл `index.html` в браузере. Будет работать импорт, история, аналитика и локальный черновик плана.

### Вариант с ИИ

1. Заполните локальный файл `conf.json`:

```json
{
  "server": {
    "host": "127.0.0.1",
    "port": 8765
  },
  "storage": {
    "databasePath": "data/training_coach.sqlite3"
  },
  "llm": {
    "provider": "openrouter",
    "apiKey": "sk-or-...",
    "apiKeyEnv": "OPENROUTER_API_KEY",
    "model": "openai/gpt-oss-120b:free",
    "fallbackModels": [
      "qwen/qwen3-235b-a22b:free",
      "deepseek/deepseek-chat-v3-0324:free",
      "meta-llama/llama-3.3-70b-instruct:free"
    ],
    "chatCompletionsUrl": "https://openrouter.ai/api/v1/chat/completions",
    "timeoutSeconds": 60,
    "temperature": 0.2,
    "maxTokens": 2500,
    "jsonMode": true,
    "siteUrl": "http://127.0.0.1:8765",
    "appName": "Training Coach"
  }
}
```

Если не хотите хранить ключ в файле, оставьте `apiKey` пустым и задайте переменную окружения с именем из `apiKeyEnv`.

Если основная модель получает `429 rate limit`, backend автоматически попробует модели из `fallbackModels` по порядку.
Если модель вернет невалидный JSON, backend также попробует следующую fallback-модель. `jsonMode` просит OpenRouter вернуть именно JSON-объект; если конкретная модель не поддерживает этот режим, его можно временно выключить.

2. Запустите локальный сервер:

```powershell
.\.venv\Scripts\python.exe server.py
```

Если Python добавлен в `PATH`, можно короче:

```powershell
python server.py
```

Если настроен Python Launcher:

```powershell
py server.py
```

PowerShell-сервер `server.ps1` оставлен как запасной вариант, но SQLite-хранилище реализовано в `server.py`.

Если все же нужен запасной PowerShell-запуск без SQLite:

```powershell
powershell -ExecutionPolicy Bypass -File .\server.ps1
```

3. Откройте в браузере адрес из `conf.json`:

```text
http://127.0.0.1:8765
```

4. На вкладке "План" нажмите "Сформировать ИИ".

API-ключ не отправляется в браузер. Он используется только локальным сервером. Файл `conf.json` добавлен в `.gitignore`; безопасный шаблон лежит в `conf.example.json`.

Тренировки и профиль сохраняются в SQLite-файл из `storage.databasePath`. По умолчанию это `data/training_coach.sqlite3`.

## Форматы импорта

Приложение не привязано к конкретному производителю. Подходят файлы, экспортированные из Garmin, Amazfit, спортивных часов, мобильных приложений и других сервисов, если они содержат поддерживаемые поля.

### TCX

Используются поля активности, кругов, длительности, дистанции и среднего пульса.

### GPX

Используются точки трека, время, дистанция по координатам и пульс, если он есть в расширениях GPX.

### CSV

CSV может быть с разделителем `,`, `;` или tab. Если файл содержит сначала сводную строку тренировки, а ниже отдельную таблицу точек/сэмплов, импортируется сводная тренировка из верхнего блока.
После импорта приложение показывает, сколько тренировок реально добавлено, сколько уже было в базе и сколько строк пропущено из-за отсутствия даты или длительности.

Поддерживаются распространенные имена колонок:

- `date`, `start time`, `start_time`, `time`, `дата`
- `sport`, `type`, `exercise`, `вид`, `спорт`
- `duration`, `duration min`, `duration_min`, `длительность`
- `distance`, `distance km`, `distance_km`, `total distance`, `total distance (km)`, `дистанция`
- `pace`, `pace (min/km)`, `avg pace`, `average pace`, `average pace (min/km)`, `avg_pace`, `темп`, `средний темп`
- `speed`, `speed (km/h)`, `avg speed`, `average speed`, `average speed (km/h)`, `avg_speed`, `скорость`, `средняя скорость`
- `avg hr`, `average heart rate`, `average heart rate (bpm)`, `avg_hr`, `hr (bpm)`, `средний пульс`
- `rpe`, `effort`
- `notes`, `comment`, `заметки`

## Как работает ИИ-план

Frontend отправляет на локальный backend:

- профиль спортсмена;
- целевую дистанцию: 5 км, 10 км, 21 км или 42 км;
- готовность и нагрузку за 7/28 дней;
- оценку текущего тренировочного состояния: acute/chronic ratio, ramp rate, частоту тренировок, время после последней тренировки, самую длинную и самую тяжелую недавнюю работу;
- последние тренировки с длительностью, дистанцией, импортированным темпом `paceMinPerKm` и пульсом, если данные есть;
- локальный черновик плана.

Backend вызывает OpenRouter Chat Completions API и просит модель вернуть JSON:

- `summary` - краткая логика плана;
- `stateAssessment` - оценка текущей формы;
- `days` - 7 дней с датой, фокусом, названием, деталями, уровнем нагрузки и обоснованием.

Prompt требует строить календарный недельный микроцикл с понедельника по воскресенье: понедельник - восстановительный бег или отдых, вторник - интервальная работа, суббота - темпо, воскресенье - длительная, остальные тренировочные дни - кроссы или восстановление по состоянию. Между интервальной и темповой работами должно быть не меньше 2 полных дней без качественных тренировок, а темпо и длительная идут связкой в выходные.
Целевая дистанция меняет содержание каждой работы: для 5 км больше коротких интервалов, скорости и VO2max; для 10 км - интервалы 800-1200 м, порог и темповая устойчивость; для 21 км - длинные интервалы, темпо и длительная аэробная устойчивость; для 42 км - контролируемые интервалы, марафонское усилие, аэробная база, длинные тренировки и восстановление.

Карточки плана подсвечиваются по факту выполнения: день считается выполненным, если в истории есть тренировка с такой же календарной датой. Будущие тренировки и текущий день выделяются отдельно.
Приложение также классифицирует выполненные тренировки как интервалы, темпо, длительную, кросс или восстановление по заметкам, длительности, дистанции, пульсу/RPE, нагрузке и разбросу скорости. Для CSV с посекундными данными анализируются 30-секундные окна скорости и пульса: повторяющиеся быстрые отрезки с восстановлением считаются признаком интервальной работы. Для марафонской цели тренировка короче 100 минут не считается длительной только по дистанции. Если ключевая работа недели уже выполнена, локальный план не дублирует ее в будущие дни: интервалы или темпо заменяются кроссом, а повторная длительная - восстановлением.

Нагрузка в истории и сводках хранится как TRIMP. Если в файле есть `Training load`, `Cardio load` или `TRIMP`, приложение берет это значение как импортную кардионагрузку. Если поле пустое, TRIMP рассчитывается по длительности, среднему пульсу, максимальному пульсу и пульсу покоя. RPE больше не используется для расчета у.е. нагрузки, но остается полезным признаком для ручной классификации тяжести тренировки.

Если сервер или API недоступны, приложение оставляет локальный план и показывает причину ошибки.
